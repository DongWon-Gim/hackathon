## 이슈 해결 기록

### 이슈 1 — useAuth SSR 하이드레이션 불일치

**증상:** 로그인 후 페이지를 새로고침하면 로그인 상태가 사라진다. 서버에서 렌더링된 HTML에는 "안녕하세요, 홍길동님"이 표시되지만, 클라이언트 JS가 마운트되면서 "로그인" 버튼으로 되돌아간다.

**원인 분석:** `useAuth.ts`에서 `user`와 `isAuthenticated`를 모듈 최상위 `ref()`로 선언했다. Nuxt SSR에서 모듈 스코프 변수는 서버 측에서 요청 간 공유될 수 있고, 클라이언트 하이드레이션 시 서버 상태와 클라이언트 상태가 일치하지 않으면 Vue가 경고를 발생시키며 클라이언트 값으로 덮어쓴다. 서버는 DB에서 가져온 사용자 정보를 갖고 있지만, 클라이언트의 초기 모듈 스코프 `ref(null)`이 이를 덮어버렸다.

**시도한 접근:**
1. `localStorage`에 사용자 정보 저장 → SSR에서 `localStorage`에 접근할 수 없어 Nuxt 빌드 오류
2. `onMounted`에서만 사용자 정보 로드 → 서버 렌더링 시 빈 상태 노출 (깜빡임 현상)

**최종 해결:** `useState()`로 교체 (`c34d339`). Nuxt의 `useState`는 서버와 클라이언트 간 상태를 동기화하는 SSR-safe 방법이다. 서버에서 설정한 상태가 클라이언트 하이드레이션 시 직렬화되어 전달되므로 불일치가 발생하지 않는다.

### 이슈 2 — 인사이트 API 경로 불일치

**증상:** 대시보드에서 "인사이트 생성" 버튼을 클릭하면 404 오류가 발생한다.

**원인 분석:** 기획 초안에서 인사이트 생성 API를 `/api/insights/generate`로 설계했다가, 구현 시 Nuxt 파일 기반 라우팅 구조상 세션 컨텍스트를 명확히 하기 위해 `/api/sessions/:id/insights`로 변경했다. 프론트엔드 코드를 수정하지 않은 채 API 경로만 바꿔서 생긴 불일치였다.

**시도한 접근:**
- API 경로를 다시 `/api/insights/generate`로 되돌리는 방안 → 세션 ID가 URL에 없어 미들웨어에서 컨텍스트 파악이 번거로움

**최종 해결:** 프론트엔드 호출 경로를 `/api/sessions/${sessionId}/insights`로 수정. 동시에 아키텍처 문서의 API 설계 표도 업데이트했다 (Sprint 2에서 전면 재작성).

### 이슈 3 — JWT secret 미설정 시 무음 실패

**증상:** `.env`에 `JWT_SECRET`을 설정하지 않은 로컬 환경에서 로그인은 성공하지만, 이후 API 호출 시 모든 요청이 401로 실패한다. 오류 메시지가 없어 원인 파악이 어렵다.

**원인 분석:** `jwt.ts`에서 `useRuntimeConfig().jwtSecret`을 직접 사용하는데, 환경변수가 없으면 `undefined`가 된다. `jsonwebtoken`의 `sign(payload, undefined)`는 에러를 던지지 않고 `undefined`를 비밀키로 사용해 토큰을 서명한다. 이후 `verify(token, undefined)` 호출 시 서명 불일치로 `JsonWebTokenError`가 발생한다. 에러는 미들웨어에서 잡히지만, 개발자는 "환경변수 누락"이라는 원인을 알 수 없다.

**시도한 접근:** `.env` 파일을 반드시 설정하도록 README에 경고 추가 → 팀원이 없는 1인 해커톤이지만 재현 가능성을 위해 코드 레벨 방어가 필요하다고 판단

**최종 해결 (코드리뷰 반영):** `jwt.ts`에서 secret이 falsy이면 즉시 예외를 던지도록 수정. `49a1a40` 커밋에서 폴백값 추가 및 개발 환경 경고 로그 포함.

## 의사결정 기록

### 결정 1 — Feedback 테이블에 userId 저장하지 않는다

**결정:** `Feedback` 모델에 `userId` 컬럼을 완전히 제외한다.

**배경/제약 조건:** RetroLens의 핵심 가치 제안은 "익명 피드백으로 솔직한 의견 수집"이다. JWT로 인증을 강제하는 구조에서 피드백과 사용자를 연결하면, 서버 측 쿼리 한 번으로 누가 무슨 피드백을 썼는지 파악할 수 있다. 팀원들이 이를 인지하면 솔직한 피드백 대신 눈치 본 피드백을 제출하게 된다.

**검토한 대안:**
- userId 저장 후 API 레이어에서 필터링 → 서버 코드 수준의 보호이므로 DB 직접 접근 시 무력화
- 암호화된 userId 저장 → 복호화 가능하므로 진정한 익명성이 아님

**채택 이유:** DB 레벨에서 연결 고리를 끊는 것이 유일하게 검증 가능한 익명성 보장 방법이다. "이 피드백이 누구 것인지 알 수 없다"는 보증을 코드 감사(audit)로 입증할 수 있다.

---

### 결정 2 — 인증 방식으로 httpOnly 쿠키 선택

**결정:** JWT를 `Authorization: Bearer` 헤더가 아닌 `httpOnly` 쿠키로 전달한다.

**배경/제약 조건:** Nuxt SSR 환경에서 서버 렌더링 시에도 인증이 필요하다. `localStorage` 기반 토큰은 SSR에서 접근 불가하고, 클라이언트 JS에서 접근 가능한 토큰은 XSS 공격에 노출된다.

**검토한 대안:**
- `localStorage` + 클라이언트 요청 시 헤더 추가 → SSR 페이지에서 인증 불가, XSS 위험
- 세션 스토어 (Redis 등) → 해커톤 환경에서 인프라 추가 불가

**채택 이유:** `httpOnly` 쿠키는 JS로 접근이 불가하여 XSS에 안전하고, 브라우저가 자동으로 쿠키를 포함하므로 SSR과 CSR 모두에서 별도 처리 없이 인증이 동작한다. Nuxt 미들웨어에서 `getCookie(event, 'auth_token')`으로 SSR 시 쿠키를 읽을 수 있다.

### 결정 3 — AI 모델로 Claude Haiku 선택

**결정:** 문체 변환과 인사이트 생성 모두 `claude-haiku-4-5-20251001`을 사용한다.

**배경/제약 조건:** 피드백 제출 시마다 AI 변환이 호출될 수 있어 응답 속도가 UX에 직접 영향을 준다. 해커톤 API 크레딧이 한정되어 있으며, 문체 변환은 창의적 생성보다 정해진 패턴 적용에 가깝다.

**검토한 대안:**
- Claude Sonnet → 품질 향상 기대, 비용 4배, 응답 속도 ~2배 느림
- Claude Opus → 불필요한 고품질, 비용 20배

**채택 이유:** 문체 변환의 핵심 요구사항은 "이모지와 개인적 말투를 제거하고 의미를 보존하는 것"이다. 이는 Haiku 수준의 언어 이해로 충분히 해결된다. 인사이트 생성도 정해진 JSON 스키마 출력이 주목적이므로 Haiku가 적합하다.
