## 이슈 해결 기록

### 이슈 1 — `readBody` 스트림 소비 순서 오류

**증상:** 투표 API(`POST /api/feedbacks/:id/vote`)에서 `body.userId`를 읽으면 `undefined`가 반환된다. Postman에서 직접 요청하면 정상 동작하는데 Vitest 통합 테스트에서만 실패한다.

**원인 분석:** Nuxt의 `readBody(event)`는 HTTP 요청의 body 스트림을 한 번만 읽을 수 있다. 미들웨어 체인에서 `auth.ts`가 JWT 검증을 위해 `event.context`를 설정하는 과정에서, 미들웨어 내부의 다른 로직이 `readBody`를 먼저 호출하고 있었다. 핸들러에서 `readBody`를 다시 호출하면 이미 소비된 스트림이라 빈 객체를 반환한다.

**시도한 접근:**
1. 미들웨어에서 `readBody` 제거 후 재테스트 → 증상 재현 안 됨을 확인
2. `event._requestBody` 캐싱 방식 검토 → h3 내부 구현에 의존하는 방법이라 취약

**최종 해결:** 핸들러 최상단에서 `readBody`를 가장 먼저 호출하도록 코드 순서 조정 (`1715366`). DB 쿼리나 권한 검증보다 body 파싱을 먼저 수행. `undefined` 반환 시 빈 객체 폴백 `const body = await readBody(event) || {}`도 함께 추가했다 (`3ee0585`).

### 이슈 2 — Claude API JSON 파싱 오류 (마크다운 코드블록)

**증상:** 인사이트 생성 API가 간헐적으로 500을 반환한다. 로그를 보면 `SyntaxError: Unexpected token '`'` — JSON.parse 실패다.

**원인 분석:** Claude에게 JSON 형식으로 응답하도록 프롬프트를 작성했지만, Claude Haiku가 가끔 JSON을 마크다운 코드블록으로 감싸서 반환한다:
```
\`\`\`json
{ "summary": "...", "issues": [...] }
\`\`\`
```
`JSON.parse`는 앞뒤의 `` ```json `` 과 `` ``` ``를 처리할 수 없어 파싱이 실패한다.

**시도한 접근:**
1. 프롬프트에 "코드블록 없이 순수 JSON만 반환하라" 명시 → 여전히 간헐적 발생
2. 응답 전체를 정규식으로 JSON 부분만 추출 → 과도한 처리

**최종 해결:** `generateInsight()`에서 응답 텍스트의 마크다운 코드블록을 제거하는 전처리를 추가 (`2f859b4`):
```typescript
const cleaned = text.replace(/^```(?:json)?\n?/, '').replace(/\n?```$/, '').trim()
JSON.parse(cleaned)
```
try/catch도 함께 감싸서 파싱 실패 시 폴백 인사이트를 반환하도록 했다.

### 이슈 3 — 공유 인사이트 `teamId` 필터 버그

**증상:** 공유 인사이트 목록 페이지(`/insights/shared`)에 자기 팀의 인사이트만 표시된다. 다른 팀이 공유한 인사이트가 보이지 않는다.

**원인 분석:** `shared.get.ts`에서 `findMany({ where: { isShared: true, session: { teamId: user.teamId } } })`로 쿼리했다. "공유" 기능의 목적은 팀 간 우수 사례 공유인데, `teamId` 필터가 이를 무력화시켰다. 코드를 작성할 때 "내 팀 인사이트만 보여주면 된다"고 잘못 이해했다.

**시도한 접근:** 필터 완전 제거가 맞는지 PRD를 재확인 → PRD: "다른 팀의 공유된 인사이트를 볼 수 있다"고 명시됨

**최종 해결:** `teamId` 필터 완전 제거. `where: { isShared: true }`만 적용 (`b95639d`). 공유 인사이트 카드에 팀명을 표시하도록 API 응답에 `team.name`도 포함시켰다.

## 의사결정 기록

### 결정 1 — AI 인사이트 폴백 전략: `buildFallback()` 함수

**결정:** Claude API 장애 시 HTTP 에러를 반환하지 않고, 피드백 데이터를 기반으로 임시 인사이트를 생성하여 반환한다. 응답에 `isFallback: true` 플래그를 포함한다.

**배경/제약 조건:** 팀장이 세션을 마감하고 인사이트를 생성하려는 순간에 Claude API가 다운되어 있으면, 회고 미팅 자체가 진행 불가한 상황이 된다. 해커톤 평가 데모 시에도 AI 서비스 장애가 발생할 수 있다.

**검토한 대안:**
- 503 Service Unavailable 반환 → 사용자 경험 최악, 데모 실패 위험
- 이전에 성공한 인사이트 캐시 반환 → 세션 첫 생성 시 캐시 없음
- 타임아웃 후 재시도 → 사용자가 기다려야 함

**채택 이유:** 피드백이 DB에 있으므로 PROBLEM 피드백 3개 + TRY 피드백 2개를 이슈로 구성하는 단순 로직으로 "쓸 만한" 임시 인사이트를 만들 수 있다. `isFallback: true` 플래그로 클라이언트가 UI에 "임시 데이터 사용" 모달을 띄워 팀장이 인지하게 했다.

---

### 결정 2 — 실시간 UX: WebSocket 대신 5초 폴링

**결정:** 피드백 입력 페이지와 대시보드의 실시간 갱신을 WebSocket이 아닌 5초 interval 폴링으로 구현한다.

**배경/제약 조건:** 해커톤 환경에서 WebSocket 서버 설정은 Vercel 서버리스와 호환되지 않는다. Vercel Functions는 요청-응답 모델만 지원하며 지속 연결이 불가하다.

**검토한 대안:**
- Server-Sent Events (SSE) → 단방향이지만 Vercel에서 스트리밍 지원 불확실
- Long polling → 서버 구현 복잡도 증가
- 클라이언트 수동 새로고침 → UX 저하

**채택 이유:** 5초 폴링은 회고 세션 컨텍스트에서 충분한 실시간성이다 (피드백을 초 단위로 확인할 필요가 없다). `useFetch`의 `refreshInterval` 옵션 한 줄로 구현되므로 복잡도가 없다. 투표 진행 중 폴링 스킵 로직 (`if (votingId.size > 0) return`)으로 UI 충돌도 방지했다.

---

### 결정 3 — 투표 낙관적 업데이트 적용

**결정:** 투표 버튼 클릭 시 서버 응답을 기다리지 않고 즉시 UI를 업데이트한다. 서버 응답 실패 시 롤백한다.

**배경/제약 조건:** 투표는 사용자 인터랙션이 즉각적인 피드백을 기대하는 행위다. 서버 왕복 시간(~300ms) 동안 버튼이 반응이 없으면 "클릭이 안 됐나?" 하고 다시 클릭하는 문제가 생긴다.

**검토한 대안:**
- 서버 응답 후 UI 업데이트 (일반 방식) → 응답성 저하, 중복 투표 시도 가능성
- 낙관적 업데이트 + 폴링 동기화 → 5초 폴링과 함께 사용하면 최종적으로 서버 상태로 수렴

**채택 이유:** 투표는 멱등성이 보장되고 (DB `@@unique` 제약), 실패율이 낮으므로 낙관적 업데이트가 적합하다. 클라이언트에서 `hasVoted` 상태를 즉시 반전시키고, 다음 폴링 사이클에서 서버 상태로 자동 동기화된다.
